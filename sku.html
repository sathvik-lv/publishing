Sales Analytics Automation (SKU-Level)

Domain-agnostic SKU analytics — demonstrated below on a jewellery dataset (154 SKUs, 10 yrs of synthetic sales, 114k transactions, 1.6k reviews + 1.3k CRE inquiries), but the upload flow at the bottom works for any retail vertical. Built on a custom statistical API stack (Bayesian, HMM, GARCH, Isotonic-calibrated ensemble), with reconciliation engine, CRE-driven recommendations, and PBT-aware profit projection. ML classifier auto-tags free-text descriptions back into SKU/category labels — retrainable on your own domain via contribute-training.

09Analyze your own data — SKU identification & HIGH/MID/LOW tiering

Drop your own sales file (CSV or XLSX) to identify SKUs and classify them as HIGH / MID / LOW performers by revenue percentile. Optionally drop opening & closing inventory files for a basic reconciliation. No projections / PBT / vendor-cost analysis — that's the daily-batch pipeline.

What we need to perform the analysis
Required — sales file:
• a SKU / item-code column — any of: sku_id, sku, item, itemcode, code
• revenue per row — any of: amount, total, revenue, line_total_inr, sales
• OR quantity + unit price (revenue is computed): qty/quantity/units × price/rate/unit_price
Optional — sales file:
• category / cat / class / department → enables category mix donut
• date / txn_date / transaction_date → enables date-range detection
For reconciliation (both optional):
• opening file with sku + opening_qty (filename containing "open")
• closing file with sku + closing_qty (filename containing "clos")

⚠ Caveats — Read These

  1. Synthetic dataset. The 154-SKU jewellery dataset is generated by a Poisson-driven simulator with explicit COVID-2020 dip and vendor-cost drift. Numbers on this page are properties of that simulator, not of any real shop — the architecture, reconciliation logic, and PBT-aware recommendations are what's being demonstrated.
  2. Drag-drop is preliminary only. Browser-side analysis identifies SKUs and tiers them by revenue percentile (top 33% HIGH, mid 33% MID, bottom 33% LOW). It does not compute PBT, vendor cost, or projection — those require the daily-batch pipeline with cost data.
  3. Privacy. Files dropped above never leave your browser. SheetJS reads them in-memory; the formatted Excel download is built client-side and triggered as a local download.
  4. Reconciliation accuracy. 100% shrinkage detection on the synthetic ground-truth set comes from the simulator capping shrink at recorded-closing >= 0; on real data this number depends on inventory-counting cadence and adjustment policies.
